## 1. 总览

整个平台围绕 **日终批处理流水线** 运转，由 Celery Beat 定时触发，配合 FastAPI 提供实时查询和操作接口。

```
┌──────────────────────────────────────────────────────────────────────────────┐
│  日终流水线 (Celery Beat 定时触发)                                            │
│                                                                              │
│  15:30          16:00          16:30          17:00          17:30           │
│  数据同步  ──→  因子计算  ──→  模型打分  ──→  风控检查  ──→  报告生成       │
│  (步骤1-5)      (步骤6)        (步骤7-8)      (步骤9-11)     (步骤12)        │
│                                                                              │
└──────────────────────────────────────────────────────────────────────────────┘
         │                                        │
         ▼                                        ▼
┌─────────────────┐                    ┌─────────────────┐
│  数据源          │                    │  组合构建        │
│  Crawler(优先)   │                    │  选股+权重+约束  │

│  AKShare         │                    │  择时信号叠加    │
└─────────────────┘                    └─────────────────┘
```

## 2. 日终流水线详解

**闭环逻辑**：先决定哪些股票可选 → 再判断哪些股票更优 → 再决定买多少 → 最后控制风险并持续监控这套逻辑有没有失效。

每天下午3点半，Celery定时任务自动触发，按顺序执行9个步骤：

| 步骤 | 名称 | 输入 | 输出 | 业务决策依据 |
|------|------|------|------|-------------|
| 1 | 股票池构建 | 最新行情/状态/财务数据 | 可交易股票池 | "谁有资格被选"——排除ST/停牌/低流动性/高风险股 |
| 2 | PIT对齐验证 | 财务数据+公告日期 | PIT合规因子数据 | 确认无前视偏差，财务数据按公告日使用 |
| 3 | 因子过滤到股票池 | 全市场因子+股票池 | 池内因子数据 | 只保留池内因子，减少噪音 |
| 4 | Regime检测 | 指数行情/宽度/波动率 | 市场状态(risk_on/trending/defensive/mean_reverting) | "当前市场适合哪类Alpha" |
| 5 | 各模块打分 | 因子数据+权重 | 4个模块分数+诊断 | 每个模块独立视角下的股票好坏 |
| 6 | 信号融合 | 模块分数+IC历史+Regime | 最终alpha分数 | 动态IC加权+Regime调权+高相关收缩 |
| 7 | 组合优化 | alpha分数+风险模型+约束 | 目标权重 | "买多少"——风险约束下的最优权重 |
| 8 | 因子健康检查 | IC/PSI/KS数据 | 告警/降权/暂停 | "这套逻辑有没有失效" |
| 9 | 存档 | 全部结果 | 目标权重+分数+regime+风险暴露 | 确保可审计、可回溯 |

整个流程全自动，任何步骤失败会自动重试，所有执行过程记录到任务日志。

### 2.1 步骤 1-5：数据同步（15:30）

**触发任务**: `app.tasks.data_sync.run_daily_sync`
**核心服务**: `app.services.data_sync_service.DataSyncService`

#### 数据源选择策略

按数据类型自动选择最优源，优先级：

| 数据类型 | 优先级 |
|---------|--------|
| 股票日线 | Crawler → Tushare → AKShare |
| 股票基础信息 | Crawler → Tushare → AKShare |
| 指数日线 | Crawler → AKShare → Tushare |
| 交易日历 | Crawler → AKShare → Tushare |
| 财务数据 | AKShare → Tushare |

#### 同步流程

```
数据源拉取 → DataNormalizer 标准化 → DataCleaner 清洗 → 入库
```

同步内容：
1. **交易日历** — 交易所开休市信息
2. **股票日线** — 全市场 OHLCV + 换手率
3. **指数日线** — 沪深300/中证500/中证1000/上证50 等
4. **财务数据** — 季报/年报核心指标
5. **数据质量检查** — 覆盖率、OHLC一致性、涨跌幅校验、交叉验证
6. **PIT管理** — 按公告日标记财务数据可见性，防止前视偏差
7. **幸存者偏差处理** — 构建历史股票池（含退市股），修正回测收益虚高

#### 质量检查项

- OHLC 逻辑一致性：`high >= max(open, close)`，`low <= min(open, close)`
- 涨跌幅校验：`pct_chg ≈ (close/pre_close - 1) * 100`
- 成交额/量/均价合理性
- 数据覆盖率 ≥ 80%
- 多数据源交叉验证

### 2.2 步骤 6：因子计算（16:00）

**触发任务**: `app.tasks.factor_calc.run_daily_factor_calc`
**核心引擎**: `app.core.factor_engine.FactorEngine`
**预处理器**: `app.core.factor_preprocess.FactorPreprocessor`

#### 因子分组（11 组）

| 分组 | 因子 | 数据依赖 |
|------|------|---------|
| 价值 | EP_TTM, BP, SP_TTM, DP, CFP_TTM | 财务 + 行情 |
| 成长 | 营收YoY, 净利润YoY, 扣非YoY, ROE YoY | 财务 |
| 质量 | ROE, ROA, 毛利率, 净利率, 流动比率 | 财务 |
| 动量 | 1M/3M/6M/12M收益, 1M反转 | 行情 |
| 波动 | 20D/60D波动率, Beta, 特质波动 | 行情 |
| 流动 | 20D/60D换手, Amihud, 零收益占比 | 行情 |
| 北向 | 净买入比, 持股5日变化, 持股占比 | 北向数据 |
| 分析师 | SUE, 评级修正, 覆盖度, 盈利惊喜 | 分析师数据 |
| 微观结构 | 大单比率, 隔夜收益, 日内/隔夜比, VPIN | 行情(分时) |
| 政策 | 政策情绪, 主题暴露度 | NLP分析 |
| 情绪 | 散户情绪, 融资余额变化, 新开户增长 | 市场数据 |

#### 预处理 Pipeline（严格遵守 ADD 3.3 节，顺序不可调换）

```
缺失值处理 → 去极值 → 中性化 → 标准化 → 方向统一
```

> **V1.2 修正**：中性化必须在标准化之前，否则残差不再服从标准正态分布。

| 步骤 | 可选方法 |
|------|---------|
| 缺失值 | 均值/中位数/行业均值/MICE插补/指数衰减前向填充 |
| 去极值 | MAD(3倍) / 分位数(1%/99%) / 自适应MAD / Isolation Forest |
| 标准化 | Z-score / 排名 / Min-Max / **逆正态秩变换**(推荐) / 稳健Z-score |
| 方向统一 | 根据因子方向定义(FACTOR_DIRECTIONS)取反 |
| 中性化 | 行业哑变量回归 / log(市值)回归 / 行业+市值双重 / 约束回归(Barra Assector) |

### 2.3 步骤 7-8：模型打分（16:30）— V1.1 优化

**触发任务**: `app.tasks.model_score.run_daily_model_score`
**核心引擎**: `app.core.model_scorer.MultiFactorScorer`

**优化点**：
- **批量查询消除N+1**：`calculate_scores()` 和 `_build_factor_ic_data()` 原每因子单独查询（26+次），优化为 `IN` 批量查询（2次）
- **前瞻收益向量化**：`_get_forward_return_data()` 原逐股票循环，优化为 `groupby().shift()` 向量化

#### 加权方法

| 方法 | 说明 |
|------|------|
| 等权 | `Score = (1/K) * Σ z_k` |
| 人工权重 | `Score = Σ w_k * z_k` |
| IC 加权 | `w_k ∝ IC_k` |
| ICIR 加权 | `w_k ∝ ICIR_k` |
| LightGBM 排序 | lambdarank 目标函数，支持单调约束 |
| Stacking 集成 | 多基模型 → 元学习器组合 |

#### 高级功能

- **分层筛选**：质量筛前50% → 估值筛前50% → 动量排序
- **滚动IC动态加权**：使用过去60期IC均值作为权重
- **对抗性验证**：检测训练/测试集分布漂移，识别失效因子

### 2.4 步骤 9：组合构建

**核心模块**: `app.core.portfolio_builder.PortfolioBuilder` + `app.core.portfolio_optimizer.PortfolioOptimizer`

#### 选股方法

- Top N 选股
- 按分位数选股
- 带行业约束选股（每行业最大持仓数）

#### 权重分配方法

| 方法 | 模块 |
|------|------|
| 等权 | PortfolioBuilder |
| 评分加权 | PortfolioBuilder |
| 风险平价(逆波动率) | PortfolioBuilder |
| 均值方差优化 | PortfolioOptimizer (scipy.optimize) |
| 最小方差 | PortfolioOptimizer |
| 最大去相关 | PortfolioOptimizer |
| Mean-CVaR | RiskModel (cvxpy) |
| HRP层次风险平价 | MultiFactorScorer |

#### 约束体系

- 单股权重上限（默认 5%）
- 行业权重上限（默认 30%）
- 换手率惩罚（L1 范数收缩）
- 市场冲击模型（Almgren-Chriss 平方根法则）

#### 调仓生成

对比当前持仓与目标组合，生成卖出/买入/调整清单，计算换手率。

### 2.5 步骤 9-11：风控检查（17:00）

**触发任务**: `app.tasks.risk_check.run_daily_risk_check`
**核心引擎**: `app.core.risk_model.RiskModel`

#### 检查项与阈值

| 检查项 | 默认阈值 | 预警级别 |
|--------|---------|---------|
| 单股集中度 | > 5% | High |
| 日VaR (95%) | > 2% | High |
| 日CVaR (95%) | > 3% | Critical |
| 最大回撤 | > 10% | Critical |
| 协方差矩阵正定性 | 最小特征值 < 1e-8 | Medium |

#### 协方差估计方法

| 方法 | 适用场景 |
|------|---------|
| 样本协方差 | 基准 |
| Ledoit-Wolf 压缩 | 高维小样本（推荐） |
| EWMA | 近期权重更大 |
| DCC-GARCH | 危机期相关性上升 |
| PCA 混合 | 结构化+特质分离 |

#### VaR 估计方法

| 方法 | 特点 |
|------|------|
| 历史模拟 | 无分布假设 |
| 参数法(正态) | 简单快速 |
| Student-t | 捕捉厚尾 |
| 蒙特卡洛 | 灵活场景 |
| 流动性调整 LVaR | 加价差+冲击成本 |

## 3. 择时引擎

**核心模块**: `app.core.timing_engine.TimingEngine`

择时引擎独立于日终流水线，可叠加到组合上调整目标仓位暴露度。

### 3.1 基础择时信号

| 信号 | 逻辑 |
|------|------|
| 均线交叉 | 短均线上穿长均线 → 看多；下穿 → 看空 |
| 市场宽度 | 上涨家数占比 > 60% → 看多 |
| 波动率 | 低波动(<12%) → 看多；高波动(>25%) → 看空 |
| 回撤控制 | 回撤超阈值 → 减仓；恢复 → 加仓 |

### 3.2 机构级扩展

| 信号 | 说明 |
|------|------|
| HMM 状态识别 | 隐马尔可夫模型识别牛/熊/震荡 |
| 北向资金择时 | 北向净买入 Z-score > 0.5 → 看多 |
| 政策日历择时 | 两会/政治局会议/中央经济工作会议 |
| 季节性择时 | 春季躁动(1-2月偏多)、Sell in May、年末效应 |
| DMA 动态模型平均 | 根据各信号近期表现动态调整权重 |
| 贝叶斯共轭更新 | Beta 分布命中率 → 后验均值作权重 |

### 3.3 信号融合

```
多信号 → 融合方法(等权投票/动量加权/贝叶斯) → 综合信号 → 目标仓位暴露度
```

## 4. 回测引擎

**核心模块**: `app.core.backtest_engine.ABShareBacktestEngine`

### 4.1 A 股交易规则

| 规则 | 实现 | 业务原因 |
|------|------|---------|
| T+1 | 当日买入部分不可卖出，原有持仓仍可卖出（精确到股数追踪） | A股制度要求今天买的明天才能卖，回测必须遵守否则收益虚高 |
| 涨跌停 | 主板10%/创业板20%/科创板20%/ST5%/北交所30% | 涨停买不进去，跌停卖不出来，很多纸面Alpha加入约束后会缩水 |
| 交易单位 | 100 股整数倍 | 不能买150股，实盘必须遵守 |
| 停牌 | 不可交易 | 停牌股无法调仓，纳入组合会卡住资金 |
| 退市整理 | 不可交易 | 极高风险，不具备选股意义 |
| 分红派息 | 现金分红+送股转增，按持股期限扣税（V1.2新增） | 分红影响净值计算，持股期限税率不同（>1年0%、>30天10%、≤30天20%） |

### 4.2 交易成本

| 项目 | 费率 |
|------|------|
| 佣金 | 万2.5（最低5元） |
| 印花税 | 千1（仅卖出） |
| 过户费 | 万0.1 |
| 滑点 | 参与率模型：`base_spread + impact * vol * sqrt(participation_rate)` |

### 4.3 机构级验证

| 方法 | 目的 | 业务含义 |
|------|------|---------|
| Walk-Forward 滚动窗口 | 避免过拟合，模拟真实模型更新 | 用前2年训练、后1季度测试，滚动验证策略在不同时期是否稳定有效 |
| 蒙特卡洛置换检验 | 检验策略统计显著性 | 打乱信号与收益对应关系1000次，如果"随机策略"也能达到类似收益，说明策略没有真正预测能力 |
| 通胀夏普比率 (DSR) | 考虑多次测试后最佳 Sharpe 的显著性 | 测试100个策略选最好的，其夏普天然偏高，DSR根据测试数量自动"打折" |
| 最小回测长度 (MinBTL) | 回测太短则 Sharpe 不可信 | (z_α / SR)²，确保回测期数满足统计显著性 |
| Bootstrap 置信区间 | 估计指标的不确定性 | 用块Bootstrap保留自相关结构，估计关键指标的95%置信区间 |

## 5.5 因子监控与告警

系统持续监控每个因子和模块的健康状态，最怕的不是短期回撤，而是**因子已经失效但系统还在照常给高权重**。

### 监控指标

| 指标 | 含义 | 告警阈值 |
|------|------|---------|
| 滚动IC | 因子值与未来收益的滚动相关性 | IC均值 < 0.02 |
| IC漂移 | 近期IC是否显著低于长期均值 | 近期IC < 长期IC的50% |
| PSI | 因子分布是否漂移 | PSI > 0.25需行动 |
| KS检验 | 当前分布与参考分布是否显著不同 | p值 < 0.05 |
| 模块相关性 | 模块间是否越来越像 | 相关系数 > 0.85 |
| 覆盖率 | 因子数据是否严重不足 | 覆盖率 < 50% |

### 告警响应动作

| 异常类型 | 响应动作 |
|---------|---------|
| IC持续低于阈值 | 降权（watch→degraded→disabled） |
| PSI > 0.25 | 标记因子分布漂移，提示人工复核 |
| 模块相关 > 0.85 | 对权重较小模块做收缩（乘以0.5） |
| 覆盖率断崖（突降>20%） | 暂停该因子 |
| 风险超阈 | 降仓 |

## 7. 性能优化体系（V1.1 新增）

### 7.1 数据库层
- **复合索引**：`stock_daily(ts_code, trade_date)` 等六个复合索引，覆盖90%+查询模式
- **迁移脚本**：`scripts/add_indexes.py`（支持 SQLite 和 PostgreSQL）

### 7.2 查询层
- **N+1消除**：批量 `IN` 查询替代逐条查询
- **向量化计算**：`groupby().shift()` 替代逐股票循环

## 8. 自然语言解读

> 以下用大白话解释整个系统"为什么这么做"，与技术规格（"是什么、怎么做"）互补。

### 8.1 一句话总结

**每天收盘后，自动从网上把全市场几千只股票的行情、财务、资金流向等数据拉下来，算出一套"打分卡"给每只股票打分，挑出得分最高的一批股票组成投资组合，同时检查这个组合风险大不大，最后生成报告。** 整个过程全自动，每天下午3点半开始，一条流水线跑完大概2小时。

### 8.2 数据怎么来、来什么

每天A股下午3点收盘，3点半系统开始从三个数据源拉数据。优先用自己写的爬虫（快、稳定、免费），爬虫挂了就用 Tushare（老牌量化数据接口，部分接口要积分），Tushare 也不行就用 AKShare（开源免费，但偶尔不稳定）。三种数据源互为备份，同一个数据如果两个源都有，还会交叉比对看有没有矛盾。

拉的数据分几类：

- **行情数据**——每只股票每天的开盘价、最高价、最低价、收盘价、成交量、成交额、换手率、涨跌幅。这是最基础的数据，后面几乎所有计算都依赖它。
- **指数数据**——沪深300、中证500、中证1000、上证50这些主要指数的日线行情。用来做基准对比，比如你的组合跑赢了沪深300多少。
- **财务数据**——季报年报里的营收、净利润、ROE、ROA、毛利率、资产负债率这些。价值因子和质量因子全靠它。
- **交易日历**——哪天开市哪天休市，用来判断什么时候该调仓、回测时跳过非交易日。

数据拉下来之后不是直接用，要先过一道质量检查：收盘价是不是比开盘价还低但标记成了最高价？涨跌幅跟收盘价/昨收价算出来对不对得上？今天的数据覆盖了多少只股票，有没有大面积缺失？这些检查不过的数据会被标记出来，防止脏数据污染后面的计算。

### 8.3 拿到数据后做了什么处理、为什么

原始数据不能直接用来选股。一只股票PE是5倍，另一只是20倍，你没法说5比20好还是差，因为它们量纲不同、分布不同、可能还有极端值。所以要做一套标准化的预处理，让所有因子变成"苹果跟苹果比"。

处理分五步，顺序不能乱：

**第一步，填缺失值。** 很多因子天然就有缺失——不是每只股票都有分析师覆盖，不是每个季度都披露研发费用。缺失本身可能就是信息（不披露研发费用的公司可能研发投入低），所以除了用行业中位数填充，还会额外创建一个"是否缺失"的二值特征。对于财务数据这种季度发布的，用指数衰减填充——离公告日越远，数据的权重越低，因为信息已经过时了。如果缺失太多（覆盖率低于80%），这个因子直接标记为不可用。

**第二步，去极值。** 一只股票因为重组停牌复牌后连续涨停，动量因子可能飙到正常值的几十倍，这种极端值会把整个横截面的分布带偏。用MAD方法（中位数绝对偏差）而不是标准差来判断，因为标准差本身就会被极端值拉大，MAD更稳健。具体来说，超过中位数±3倍MAD的值会被截断到边界，而不是直接删掉——删掉会改变样本量，截断保留了信息但限制了影响。

**第三步，标准化。** 去完极值后，每只股票的因子值还是"原始量级"——PE是5-100的范围，ROE是-50%到+50%的范围。要把它们统一到同一个尺度上才能加权求和。默认用逆正态秩变换——先把所有股票按因子值排名，再把排名转换成标准正态分布的分位数。这比简单的Z-score更好，因为Z-score假设数据接近正态分布，但因子值经常是偏态的，秩变换不依赖分布假设，是学术界和AQR的标准做法。

**第四步，方向统一。** 有些因子越大越好（ROE越高公司质量越好），有些越小越好（波动率越低越好）。统一方向后，所有因子都是"分数越高，预期未来表现越好"，后面加权的时候不用再操心方向问题。

**第五步，中性化。** 这是最关键的一步。如果不做中性化，你选出来的"好股票"可能只是"某个行业的好股票"——比如银行股PE普遍低，价值因子天然就高，你的组合会严重偏向银行。中性化就是用行业哑变量做回归，取残差——残差是剔除了行业效应之后的"纯因子暴露"。同时还会对log(市值)做回归，剔除大市值/小市值的系统性偏差。这样选出来的股票是"在同一个行业内、同等市值水平下，因子值真正突出的"，而不是"恰好处于某个行业或某个市值区间"的。

### 8.4 处理完的数据怎么变成选股信号

经过上面五步处理，每只股票在每个因子上都有一个标准化的分数。但一个因子不够——只看估值可能买到"便宜但有原因"的垃圾股，只看动量可能追高被套。所以要把多个因子综合成一个总分。

最简单的方式是等权平均——所有因子分数加起来除以因子个数。但不同因子的预测能力不一样，IC（因子值和未来收益的相关性）高的因子应该给更大权重。所以更合理的做法是ICIR加权——用每个因子的IC均值除以IC标准差作为权重，IC高且稳定的因子权重大，IC波动大的因子会被惩罚。

如果还想更精细，可以用LightGBM把选股建模成一个排序问题——让模型自己学习哪些因子在什么条件下更重要，还能加入单调约束（比如ROE这个因子，不管什么情况下分数越高都不应该导致排名下降）。

还有一种分层筛选的思路：先按质量因子筛掉后50%的公司（质量不过关的再便宜也不买），再按估值因子筛掉后50%（质量过关但太贵的不买），最后按动量排序（质量好、估值合理、近期有资金关注的优先）。这比一次性加权更符合投资逻辑。

### 8.5 有了总分之后怎么构建组合

给几千只股票打完分，选出得分最高的50只（或100只），接下来要决定每只买多少。

最简单的是等权——每只买2%（50只各2%）。但等权有个问题：如果某只股票波动率特别大，它对组合风险的贡献远超2%，相当于你给了高风险股票过高的风险预算。风险平价就是反过来——波动率高的股票给小权重，波动率低的给大权重，让每只股票对组合风险的贡献相等。

更精细的做法是均值方差优化——输入每只股票的预期收益和它们之间的协方差矩阵，在"总权重=100%"和"单只不超过5%"的约束下，找到风险调整后收益最高的权重组合。协方差矩阵的估计本身是个难题，样本协方差在股票数量多、历史数据短的时候不稳定，所以用Ledoit-Wolf压缩估计——把样本协方差往一个结构化目标（比如缩放后的单位矩阵）上"拉"一点，拉多少由数学公式自动决定，在偏差和方差之间取最优平衡。

组合建好后还要加约束：单只股票不超过5%（集中度风险）、单个行业不超过30%（行业暴露风险）、换手率不能太高（交易成本会吃掉收益）。

### 8.6 择时——什么时候该满仓什么时候该减仓

上面选股和组合构建解决的是"买什么、买多少"的问题，但没解决"现在该不该买"的问题。如果大盘要跌，好股票也会跌。择时引擎就是判断市场整体环境的。

它看几个信号：均线交叉（短期均线上穿长期均线说明趋势向上）、市场宽度（全市场上涨家数占比，如果绝大多数股票都在涨说明是系统性机会而不是个别股票异动）、波动率（低波动环境适合持有，高波动环境适合减仓）、回撤控制（如果组合已经从高点回撤超过10%，强制减仓止损）。

A股还有一些特有的择时信号：北向资金（外资是A股最有效的聪明钱信号之一，北向持续净买入通常预示市场走强）、政策日历（两会、政治局会议这些重要时间窗口前后市场往往有规律性表现）、季节性（1-2月春季躁动偏多，5-9月Sell in May偏空，12月基金排名效应偏多）。

多个择时信号之间可能矛盾——均线说看多，波动率说看空。用贝叶斯共轭更新来融合：每个信号维护一个Beta分布的"命中率"，近期预测准的信号权重自动上升，预测差的自动下降，不用人工调参。

择时的最终输出是一个0到1的"目标仓位暴露度"——1表示满仓，0.5表示半仓，0表示空仓。这个数字乘到组合权重上，就实现了"选股决定买什么，择时决定买多少"。

### 8.7 风控——组合建好了，风险大不大

组合建好后，每天收盘会跑一轮风控检查。

最核心的是VaR（Value at Risk）——"在最差的5%的情况下，这个组合一天最多亏多少"。如果算出来是2%，意思是95%的交易日亏损不会超过2%。但VaR只告诉你"最差情况的门槛"，不告诉你"过了这个门槛之后亏多少"。CVaR（Conditional VaR）解决这个问题——它算的是"在最差的5%的情况里，平均亏多少"，永远比VaR大，是对尾部风险更诚实的度量。

A股的收益率分布不是正态的——有肥尾（极端事件比正态分布预测的更频繁），所以用Student-t分布来估计VaR比正态假设更准确。还会算流动性调整VaR——如果你的持仓金额占日成交额的比例很高，真要卖的时候价格会被自己砸下去，这个冲击成本要加到风险里。

如果风控检查触发了预警（比如CVaR超过3%、回撤超过10%），会按严重程度分级记录——Critical级别的预警会触发日志告警，需要人工介入。

### 8.8 回测——这套东西到底管不管用

所有策略在实盘之前必须回测。回测引擎模拟历史交易，看如果过去真的按这套逻辑操作，收益和风险会怎样。

回测不是简单地"买入持有算收益"，要严格遵守A股的交易规则：T+1（今天买的明天才能卖）、涨跌停（涨停板买不进去，跌停板卖不出来）、100股整数倍（不能买150股）、停牌（停牌的股票不能交易）。还要扣除真实成本——佣金万2.5、印花税千1（卖出时收）、过户费、滑点（你的买单会推高成交价，卖单会压低成交价，大单的冲击更大）。

回测最容易犯的错是过拟合——在历史数据上表现完美，实盘一塌糊涂。所以要做Walk-Forward验证：用前2年数据训练，后1个月测试，然后滚动窗口，看策略在不同时期的表现是否稳定。还要做蒙特卡洛置换检验——把选股信号和收益率的对应关系打乱重排1000次，如果打乱之后的"随机策略"也能达到差不多的收益，说明你的策略没有真正的预测能力，只是碰巧了。通胀夏普比率（DSR）更进一步——如果你测试了100个策略选了最好的那个，它的夏普比率天然就偏高，DSR会根据你测试的策略数量自动"打折"，只有打折后仍然显著的才是真正有效的。

### 8.9 整个流程怎么运转

每天下午3点半，Celery定时任务自动触发，按顺序执行：数据同步 → 因子计算 → 模型打分 → 风控检查 → 报告生成。每个步骤如果失败会自动重试3次，所有执行过程记录到任务日志表。

用户也可以随时通过API或前端手动触发某个步骤、调整模型参数、查看结果。比如研究员想测试一个新的因子组合，可以手动配置一个新模型，指定用哪些因子、怎么加权、选多少只股票，然后跑一次回测看效果。

最终产出的，是每天一份"应该买什么、买多少、为什么"的投资组合建议，附带风险检查报告和绩效归因分析，可以直接用于实盘交易或提供给订阅客户。
